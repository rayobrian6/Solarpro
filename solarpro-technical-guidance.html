3D Render Performance Optimization
CesiumJS entity-based rendering · requestRenderMode: true · Google Photorealistic 3D Tiles
Architecture context (from codebase analysis): Current pipeline: panels prop change → useEffect → renderAllPanels() → remove ALL entities → rebuild ALL entities. Root bottleneck: O(n) full rebuild on every panel change, even when only 1 panel was added. requestRenderMode: true is already enabled (good baseline — GPU only renders on demand).
🔴 Priority 1 — High Impact
1. Incremental Diff Rendering
✅ Done in v30.5 95-99% fewer entity ops

What was implemented: renderAllPanels() now diffs lastRenderedPanelsRef vs new panel list. Only removes deleted panels, only adds new panels, only rebuilds panels whose geometry changed.

  • Before: 200 entity removes + 201 entity adds = 401 Cesium operations for adding 1 panel
  • After: 1 entity add = 1 Cesium operation

Remaining opportunity: Panel position updates currently remove+re-add the entity. For drag operations, update entity.position and entity.orientation in-place:

// Instead of remove + re-add for position changes: function updatePanelEntityInPlace(entity, C, panel) { const pos = safeCartesian3(C, panel.lng, panel.lat, panel.height); const hpr = new C.HeadingPitchRoll(panel.heading, -panel.tilt * Math.PI/180, 0); const orientation = C.Transforms.headingPitchRollQuaternion(pos, hpr); entity.position = new C.ConstantPositionProperty(pos); entity.orientation = new C.ConstantProperty(orientation); // No entity remove/add — eliminates flicker during drag }
Impact
97%↓ ops
2. Debounced Batch Rendering
✅ Done in v30.5 Eliminates redundant rebuilds

What was implemented: 16ms debounce on the panels useEffect batches rapid state changes (auto-fill, undo/redo) into a single render call.

Remaining opportunity: Dynamic debounce window based on panel count delta:

const delta = Math.abs(panels.length - lastRenderedPanelsRef.current.length); const debounceMs = delta > 20 ? 50 : delta > 5 ? 32 : 16; // 50ms for 200-panel auto-fill, 16ms for single panel clicks

3. Cesium Primitive-Based Rendering (GPU Instancing)
Moderate — 2 days 60-80% GPU reduction

Current problem: viewer.entities.add() — each panel is a separate JavaScript object with full Cesium property system overhead. Cesium entities are designed for data-driven visualization, not high-frequency geometric rendering.

Solution: Replace with GeometryInstance + Primitive — single GPU draw call for ALL panels:

function buildPanelPrimitive(C, panels) { const instances = panels.map(panel => { const pos = C.Cartesian3.fromDegrees(panel.lng, panel.lat, panel.height); const hpr = new C.HeadingPitchRoll(panel.heading, -panel.tilt * Math.PI/180, 0); const modelMatrix = C.Transforms.headingPitchRollToFixedFrame(pos, hpr); C.Matrix4.multiplyByScale(modelMatrix, new C.Cartesian3(PW, PH, PT), modelMatrix); return new C.GeometryInstance({ geometry: new C.BoxGeometry({ vertexFormat: C.PerInstanceColorAppearance.VERTEX_FORMAT }), modelMatrix, attributes: { color: C.ColorGeometryInstanceAttribute.fromColor(systemTypeColor(C, panel.systemType)) }, id: panel.id, // for picking }); }); return new C.Primitive({ geometryInstances: instances, appearance: new C.PerInstanceColorAppearance({ flat: false }), releaseGeometryInstances: false, // keep for picking allowPicking: true, asynchronous: true, // compile on worker thread }); }
  • Single draw call for ALL panels (GPU instancing) vs. one draw call per panel
  • No JS property system overhead per panel — pure geometry
  • Shadow rendering: one shadow pass for all panels vs. n passes
  • Memory: ~70% reduction (no per-entity JS objects)
  • Color updates: primitive.getGeometryInstanceAttributes(id).color = newColor — in-place, no rebuild
GPU Impact
60-80%↓

5. Level of Detail (LOD) — Distance-Based Quality Switching
Moderate — 1 day 40-60% GPU at overview

At camera distances > 500m, individual panel geometry is sub-pixel. Rendering full box geometry wastes GPU cycles.

B. Panel Arrangement Algorithm

⚡ Inter-Row Spacing Formula (Industry Standard — Winter Solstice)
/** * Minimum row spacing to prevent inter-row shading * Based on winter solstice sun angle (worst case for shading) */ function calcMinRowSpacing(tiltDeg, panelHeightM, latitudeDeg) { const tiltRad = tiltDeg * Math.PI / 180; const panelVerticalHeight = panelHeightM * Math.sin(tiltRad); // vertical rise const panelHorizontalDepth = panelHeightM * Math.cos(tiltRad); // horizontal footprint // Winter solstice sun elevation = 90° - |latitude| - 23.45° const sunElevDeg = Math.max(10, 90 - Math.abs(latitudeDeg) - 23.45); const shadowLength = panelVerticalHeight / Math.tan(sunElevDeg * Math.PI / 180); return (panelHorizontalDepth + shadowLength) * 1.1; // +10% buffer } // Example: 25° tilt, 1.722m panel, 38° latitude (Illinois) // → panelVerticalHeight = 0.728m // → sunElevation = 28.55° // → shadowLength = 1.333m // → rowSpacing = (1.560 + 1.333) × 1.1 = 3.18m ≈ 10.4 ft

C. Constraints & Validation Rules

Minimum Area

1 panel minimum: 1.434m × 2.222m ≈ 3.2 m²
Practical minimum: 10 m² (2–3 panels with spacing)
Anything smaller is not economically viable for ground mount.

Maximum Area (Performance)

Soft limit: 500 panels — warn, offer to split
Hard limit: 1000 panels — prevent browser freeze
With primitive rendering: hard limit → 5000+

Terrain Slope

< 5%: No warning
5–15%: ⚠️ Row spacing adjusted
15–25%: ⚠️ Structural review required
> 25%: ❌ Exceeds standard ground mount limit

Shape Validation (4-Point)

Check for self-intersection (bowtie) → error with diagram
Check for concavity → warning, panels near indentation excluded
Check minimum area → error with specific dimensions needed
